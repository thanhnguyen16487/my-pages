Chương 01 ● Beginner

Foundations — Harness Engineering Là Gì?

Harness engineering là ngành thực hành thiết kế môi trường xung quanh AI agents để chúng hoạt động đáng tin cậy. Nó nằm ở giao điểm của context engineering, evaluation, observability, orchestration, safe autonomy và software architecture.

Thuật ngữ "harness" mượn từ thế giới ngựa: con ngựa mạnh mẽ nhưng vô dụng cho canh tác nếu không có bộ dây cương. AI model là con ngựa. Harness là mọi thứ hướng sức mạnh của nó vào công việc hữu ích. Kỹ sư là người cưỡi, cung cấp phương hướng.

❓ WHY — Tại sao harness engineering xuất hiện ngay bây giờ?

Agents đã đủ giỏi để hữu ích, nhưng chưa đủ tin cậy để tin tưởng một mình. Khoảng cuối 2025, Andrej Karpathy mô tả sự đảo ngược: từ "chủ yếu code thủ công + ít AI hỗ trợ" sang "chủ yếu agent-driven + con người chỉnh sửa". Nhưng ngay khi agents đủ mạnh để làm nhiều, mọi người cũng phát hiện chúng dễ vỡ thế nào trong môi trường thực.

Kết quả yếu từ coding agents thường là VẤN ĐỀ HARNESS, không phải vấn đề model. Hai team dùng cùng model Claude/GPT có thể thấy chênh lệch 40 điểm phần trăm completion rate — hoàn toàn do harness design khác nhau.

Phân biệt 3 cấp độ engineering

Prompt Engineering

Tối ưu chỉ thị cho MỘT lần tương tác với model. "Hỏi cái gì" — đầu vào trực tiếp.

Context Engineering

Quản lý tập token xuyên suốt các lượt trong MỘT context window. "Gửi cho model cái gì" để nó trả lời tự tin.

Harness Engineering

TOÀN BỘ hệ thống vận hành: tools, permissions, state, tests, logs, retries, checkpoints, guardrails, reviews, evals — XUYÊN SUỐT nhiều context windows.

Context engineering nằm BÊN TRONG harness engineering. Harness bao trùm context engineering cùng với tool orchestration, state management, verification, human-in-the-loop controls, và lifecycle management.

Nếu prompt engineering là lệnh "rẽ phải", thì harness engineering là con đường, rào chắn, biển báo, và hệ thống giao thông cho phép mười phương tiện di chuyển an toàn cùng lúc.

— So sánh phổ biến trong cộng đồng harness engineering

Chương 02 ● Beginner

Agent = Model + Harness — Kiến Trúc Cơ Bản

Model là engine. Harness là phần còn lại của chiếc xe: hệ thống lái, phanh, làn đường, lịch bảo trì, đèn cảnh báo. Nếu chỉ tập trung vào engine và nhiên liệu, bạn vẫn có thể xây một chiếc xe tệ.

Harness bao gồm những gì?

Harness là toàn bộ runtime environment quản lý cách agent thực thi. Claude Agent SDK, OpenAI Codex harness, và các custom orchestration layers đều là harnesses. Chúng quản lý context, thực thi safety, xử lý failures, duy trì state, và kiểm soát lifecycle của agent.

❓ WHY — Tại sao không chỉ dùng framework?

Framework dùng để BUILD agent. Harness dùng để RUN agent. Nhiều team bỏ qua harness — deploy agents xây bằng framework trực tiếp vào production mà không có runtime infrastructure đảm bảo reliability. Đó là lúc failures bắt đầu. Một team đã deploy agent "crush" mọi demo, nhưng lúc 3 giờ sáng thứ Ba, agent đó vào retry loop với một flaky API, đốt cháy toàn bộ budget tokens.

Vòng lặp cốt lõi: ReAct Loop

Pattern phổ biến nhất cho execution là ReAct (Reasoning + Acting). Model nhận trạng thái hiện tại → suy luận bước tiếp theo → thực hiện action qua tool call → quan sát kết quả → lặp lại cho đến khi gặp stopping condition. Harness orchestrate vòng lặp này; model suy luận và chọn actions.

ReAct Loop Example User: "Fix the failing test" → Model reads test output → Reasons: import path is wrong → Action: edits file via tool → Observes: re-runs tests, new type mismatch → Action: fixes type → Observes: tests pass → Reasons: job done → STOP

Harness vs Orchestrator

Orchestrator = bộ não: phân rã goal thành sub-tasks, quyết định sequence, điều khiển reasoning loop (ReAct, tree-of-thought). Tập trung vào LOGIC và CONTROL FLOW.

Harness = đôi tay và hạ tầng: cung cấp tools, quản lý I/O, duy trì state, enforce safety. Tập trung vào CAPABILITIES và SIDE-EFFECTS. Cải thiện ở bất kỳ bên nào đều cải thiện dramatically performance tổng thể.

Chương 03 ● Intermediate

6 Components Cốt Lõi Của Agent Harness

Sáu thành phần này quyết định agent của bạn thành công hay thất bại — không phải model bạn chọn.

📋
1. Context Engineering

Quản lý context window như ngân sách working memory — không phải thùng rác. Quyết định instructions, evidence, intermediate artifacts, và state nào nên có ở mỗi bước.

🔧
2. Tool Orchestration

Registry tools, dispatch, routing. Bash + code execution là general-purpose strategy. Ít tools hơn có thể cho kết quả TỐT hơn (Vercel bỏ 80% tools → kết quả tốt hơn).

💾
3. State Management

Duy trì state xuyên sessions: filesystem, git history, progress files, feature lists. Mỗi session mới bắt đầu với ZERO memory — harness phải bridge the gap.

✅
4. Verification Loops

Write → Test → Fix cycles. Agent tự verify output trước khi declare done. Self-verification loops là harness change tạo cải thiện lớn nhất.

🛡️
5. Human-in-the-Loop

Thiết kế KHI NÀO và THẾ NÀO con người được tham vấn. Full autonomy hiếm khi phù hợp. Approval flows, review checkpoints, escalation paths.

📊
6. Observability & Lifecycle

Log mọi action, track token usage, record decision points, surface anomalies. Đây là ranh giới giữa research prototype và production system.

LangChain's coding agent nhảy từ 52.8% lên 66.5% trên Terminal Bench 2.0 — chỉ bằng cách thay đổi harness, không đổi model. Thêm self-verification loop và loop detection biến performer trung bình thành top 5.

— LangChain, "Improving Deep Agents with Harness Engineering"
Chương 04 ● Intermediate

Context Engineering — Quản Lý Working Memory

Context window là tài nguyên khan hiếm. Một file instruction khổng lồ lấn chiếm task, code, và docs liên quan — agent bỏ lỡ constraints hoặc tối ưu cho thứ sai.

Nguyên tắc: Map, không phải Manual 1000 trang

OpenAI học được bài học sớm nhất: cho Codex một BẢN ĐỒ, không phải cuốn sách hướng dẫn khổng lồ. AGENTS.md là mục lục; knowledge base thực sự nằm trong thư mục docs/ có cấu trúc, được version control, và cập nhật tự động.

❓ WHY — Tại sao quá nhiều context là vấn đề?

Crowding: Instruction file lớn đẩy hết chỗ cho task, code, và relevant docs.

Non-guidance: Khi mọi thứ đều "quan trọng", không gì quan trọng. Agent pattern-match locally thay vì navigate intentionally.

Rot: Manual monolithic thành nghĩa trang rules cũ. Agent không biết cái gì còn đúng.

Hard to verify: Một blob duy nhất không phù hợp cho mechanical checks (coverage, freshness, ownership).

Patterns từ Manus (Context Engineering for AI Agents)

KV-cache locality: Tối ưu cách sắp xếp context để tận dụng KV-cache, giảm latency.

Tool masking: Chỉ hiển thị tools relevant cho bước hiện tại — giảm confusion.

Filesystem as memory: Dùng filesystem làm bộ nhớ dài hạn thay vì nhồi nhét vào context window.

Keeping useful failures in-context: Giữ lại thông tin về lỗi hữu ích để agent không lặp lại sai lầm.

Context-Efficient Backpressure

Ngăn agent đốt context vào công việc noisy hoặc low-value. Khi output từ tool quá dài hoặc không liên quan, harness cần cơ chế "back-pressure" — cắt giảm, tóm tắt, hoặc defer — để bảo vệ context budget cho công việc thực sự quan trọng.

Progressive Disclosure Pattern

Agent bắt đầu với entry point nhỏ, ổn định và được HƯỚNG DẪN nơi tìm tiếp — thay vì bị overwhelm ngay từ đầu. OpenAI áp dụng cơ chế này: AGENTS.md là bảng mục lục, liên kết tới docs/ cụ thể. Linters và CI jobs validate rằng knowledge base luôn up-to-date, cross-linked, và structured đúng. Một "doc-gardening" agent quét tìm docs lỗi thời và mở fix-up PRs tự động.

Chương 05 ● Intermediate

Constraints, Guardrails & Safe Autonomy

Câu hỏi không phải CÓ NÊN constraint agent behavior hay không — mà là LOẠI constraint nào giải quyết failure mode cụ thể.

3 loại Controls: Guides, Sensors, và Enforcement

Guides (Feedforward Controls)

Dự đoán hành vi agent và hướng dẫn TRƯỚC KHI nó hành động. Tăng xác suất kết quả tốt ngay lần đầu. Ví dụ: rules files, coding conventions, architectural specs, AGENTS.md.

Sensors (Feedback Controls)

Quan sát SAU KHI agent hành động và giúp nó tự sửa. Mạnh nhất khi tạo tín hiệu tối ưu cho LLM consumption — ví dụ custom linter messages bao gồm hướng dẫn tự sửa (một dạng "positive prompt injection"). Thiếu guides → agent lặp lỗi. Thiếu sensors → agent không biết rules có hoạt động không.

Deterministic Enforcement (CI Layer)

Tất cả rules set thành "error", KHÔNG PHẢI "warn" — chúng hoạt động như hard gates, không phải advisory signals. Linter, type checker, format checker phải CHẶN commit nếu violate, không phải nhắc nhở.

2 execution types

Computational (CPU): Tests, linters, type checkers, structural analysis. Chạy trong milliseconds đến seconds. Kết quả deterministic, đáng tin cậy.

Inferential (GPU/NPU): Semantic analysis, AI code review, "LLM as judge". Chậm hơn, đắt hơn, non-deterministic. Nhưng cho phép guidance phong phú và judgment ngữ nghĩa sâu.

❓ WHY — Tại sao Vercel bỏ 80% tools lại tốt hơn?

Nhiều tools = nhiều confusion, nhiều incorrect tool selections, nhiều failed tasks. Tool orchestration layer phải CONSTRAINT options của model để cải thiện output. Ít capability hơn, nhiều reliability hơn. Đây là insight cốt lõi của harness engineering: hạn chế có chủ đích tạo ra kết quả tốt hơn.

Rules Files & AGENTS.md

Rules files là instruction sets repository-scoped, persistent, được inject vào context agent ở đầu mỗi session. Khác biệt cơ bản so với prompts đơn giản: survive across sessions, inject tự động bởi runtime, scope toàn directory tree, compose hierarchically. AGENTS.md spec (released 08/2025) là open standard cross-tool convention — OpenAI dùng 88 files AGENTS.md xuyên suốt monorepo.

❓ WHY — Tại sao spec-driven thay vì "just prompt"?

Một feature description viết tốt trong Jira với scope rõ ràng, acceptance criteria cụ thể, và context liên quan → trực tiếp cải thiện plans và code agent tạo ra. "Same structure in, structure out" — chất lượng đầu vào quyết định chất lượng đầu ra. Khi không có spec rõ ràng, agent "đoán" về codebase thay vì "nhìn" vào nó → hallucinate file paths, invent APIs không tồn tại, modify sai module.

Feature List Pattern (Anthropic)

Cho long-running projects, initializer agent tạo file feature list toàn diện mở rộng từ prompt ban đầu. Tất cả features ban đầu marked "failing". Coding agents chỉ được thay đổi status "passes" — KHÔNG ĐƯỢC xóa hoặc edit test descriptions. Dùng JSON thay vì Markdown vì model ít likely thay đổi JSON files inappropriately.

Feature List Entry (JSON) { "category": "functional", "description": "New chat button creates fresh conversation", "steps": [ "Navigate to main interface", "Click 'New Chat' button", "Verify new conversation created", "Check chat area shows welcome state", "Verify conversation appears in sidebar" ], "passes": false }
Chương 07 ● Advanced

Evals & Observability

Bạn không thể cải thiện cái bạn không thể nhìn thấy. Observability là ranh giới giữa research prototype và production system.

Evaluation: Đo gì khi agents có nhiều trajectories?

Agents không có một đường đi duy nhất đến thành công — chúng có thể giải quyết cùng task theo nhiều cách khác nhau. Eval design phải phản ánh điều này.

📝
Single-step Evals

Đánh giá mỗi quyết định riêng lẻ: tool call đúng không? Reasoning logic không?

🔄
Full-run Evals

Đánh giá kết quả cuối cùng: task có hoàn thành? Output có correct? Bao nhiêu attempts?

🗣️
Multi-turn Evals

Đánh giá conversations dài: agent có maintain coherence? State có được tracked đúng?

Infrastructure Noise — Anthropic's Warning

Runtime configuration có thể dịch chuyển benchmark scores NHIỀU HƠN so với khoảng cách giữa các vị trí trên leaderboard. Điều này có nghĩa: nếu bạn so sánh model A vs B nhưng harness khác nhau, bạn đang so sánh harnesses, không phải models. Eval phải kiểm soát biến harness.

Observability Stack

Good observability trả lời: Tại sao agent chọn tool này? Bao nhiêu attempts để pass test suite? Ở đâu trong workflow tốn nhiều tokens nhất? Khi nào agent lần cuối cần human intervention?

Công cụ observability phổ biến:

Langfuse — self-hostable LLM observability, trace mọi agent step, quản lý prompt versions, run evals. Arize Phoenix — self-hostable trace UI, replay mọi reasoning step offline. Braintrust — evaluation-first platform, auto-tracing mọi LLM call/tool invocation. W&B Weave — tracing + eval cho agent workflows. OpenTelemetry GenAI Conventions — standard attribute names cho portable harness traces.

Harness-First Engineering (Datadog)

Datadog đề xuất: thay vì đọc mọi dòng code agent tạo ra, đầu tư vào automated checks nói cho bạn trong vài giây liệu code đúng hay không. Agent generates code → harness verifies → production telemetry validates → nếu sai, feedback cập nhật harness và agent thử lại. Đầu tư vào harness tỷ lệ thuận với chi phí failure.

Long-Running Agents — Xuyên Nhiều Context Windows

Thách thức cốt lõi: agents phải làm việc trong discrete sessions, và mỗi session mới bắt đầu với ZERO memory. Hãy tưởng tượng dự án phần mềm mà mỗi kỹ sư ca mới đến với không một ký ức về ca trước.

Initializer Agent (session đầu tiên): Prompt đặc biệt yêu cầu setup môi trường: tạo init.sh script, claude-progress.txt log, feature list toàn diện, và initial git commit.

Coding Agent (mọi session sau): Bắt đầu bằng đọc progress file + git log → chạy basic test → chọn 1 feature priority cao nhất chưa done → implement → test end-to-end → commit + update progress.

Ralph Loop Pattern

Khi agent cố exit, harness intercept exit attempt qua hook, reinject original prompt vào CLEAN context window mới, buộc agent tiếp tục làm việc against completion goal. Filesystem cho phép điều này: mỗi iteration bắt đầu fresh context nhưng ĐỌC state từ iteration trước qua file.

❓ WHY — Tại sao incremental progress quan trọng đến vậy?

Mỗi context window là hữu hạn. Nếu agent cố build toàn bộ app trong 1 window → hết context giữa chừng → session sau phải ĐOÁN chuyện gì đã xảy ra → lãng phí tokens "dọn dẹp" thay vì "xây mới". Incremental = mỗi session tạo VALUE NHỎ NHƯNG HOÀN CHỈNH → không bao giờ rơi vào trạng thái hỏng.

Chương 09 ● Advanced

Multi-Agent Architecture & Orchestration

Khi task đủ phức tạp, một single agent không còn đủ. Multi-agent systems phân chia vai trò — nhưng tạo ra challenges mới về coordination, state sharing, và verification.

Coordinator → Implementor → Verifier Pattern

Coordinator: Phân tích codebase, soạn living spec (goals, tasks, decisions, trade-offs), delegate cho Implementor agents song song.

Implementor: Thực thi tasks cụ thể trong parallel, mỗi agent với context window riêng.

Verifier: Đóng vòng lặp — kiểm tra implementation against spec, validate cross-service dependencies, flag violations TRƯỚC KHI code đến PR stage. Khi Verifier reject, rejection trở thành structured context cho correction — KHÔNG bị drop im lặng.

Anthropic's Multi-Agent Research System

Architecture với separation of roles và structured coordination. Mỗi agent có vai trò chuyên biệt, giao tiếp qua artifacts rõ ràng, và harness đảm bảo không agent nào đi chệch khỏi phạm vi được giao.

Shared Context Problem

❓ WHY — Tại sao multi-agent khó hơn single agent?

Mỗi agent có context window riêng → không tự động biết agent khác đã làm gì. Filesystem trở thành "shared ledger" — nơi agents ghi lại và đọc work progress. Nhưng conflicts xảy ra: hai agents sửa cùng file, hoặc agent A phụ thuộc vào output chưa hoàn thành của agent B. Harness phải quản lý concurrency, dependencies, và conflict resolution.

Mọi bài học harness engineering đều đến từ nỗi đau — từ người nhận ra rằng khi model đủ capable, bottleneck không còn là "có thể generate code không?" mà trở thành "có thể khiến nó behave reliably trong real system không?"

Observability for agents: Logs, metrics, traces exposed qua local observability stack. Agent query logs với LogQL, metrics với PromQL. Prompts như "ensure service startup under 800ms" trở nên tractable. Single Codex runs thường chạy 6+ giờ (thường khi con người đang ngủ).

Anthropic: Claude.ai Clone Demo

Kết quả thực tế

Initializer agent tạo 200+ features cho claude.ai clone. Mỗi coding session chọn 1 feature, implement, test qua Puppeteer MCP (screenshots thực), commit, update progress. Vấn đề còn tồn tại: Claude không thấy browser-native alert modals qua Puppeteer → features dùng modals buggier hơn. Nhưng tổng thể, harness approach biến "prompt đơn giản" thành "incremental progress đáng tin cậy".

Datadog: Harness-First Engineering

redis-rust & Helix

Harness đủ mạnh để THAY THẾ code review làm nguồn chính cho correctness. redis-rust đạt production-like staging với latency tương đương và giảm 87% memory. Helix (Kafka-compatible streaming engine) chịu hàng triệu deterministic simulation runs, đạt ~93% peak disk throughput mà không hy sinh Kafka-semantic guarantees. Nguyên tắc: đầu tư thời gian vào strengthening harness — tighten invariants, expand simulation coverage, connect telemetry feedback loops — thay vì review mọi diff.

Red Hat: Structured Workflows

Hai kỹ thuật hoạt động

1. Repository Impact Map: Trước khi agent plan, phân tích symbols thực tế trong code → ground mọi downstream task trong real code, không phải hallucination.

2. Structured Constraints: File paths, symbol names, existing patterns to follow — càng cụ thể input, càng consistent output.

Nguyên tắc: Treat harness as software you maintain. Skills, prompts, MCP configs là CODE — version, review trong PRs, refactor khi drift. Stale prompt rots giống stale test.

Martin Fowler / Thoughtworks: Cybernetics Framing

Feedforward + Feedback = Cybernetic System

Con người mang "organisational alignment" mà agent không có: awareness about team goals, technical debt nào được tolerate, "good" trông như thế nào trong context cụ thể. Agent không có social accountability, không có aesthetic disgust trước function 300 dòng, không biết convention nào load-bearing và convention nào chỉ là thói quen.

Harness là nỗ lực externalize và make explicit những gì developer experience của con người mang lại — nhưng chỉ đi được tới một mức. Mục tiêu: harness không loại bỏ hoàn toàn human review, mà giảm review toil và tăng system quality.

Tương Lai & Open Questions

Tương lai không phải "một genius model làm mọi thứ" mà là "models vận hành bên trong well-engineered environments khiến chúng sử dụng được." Và đó chính xác là lý do harness engineering quan trọng.

Harness Layer Framework (3 tầng)

Nghiên cứu học thuật đang formalize harness thành 3 tầng rõ ràng:

🎛️
Control Layer

Cách work được phân rã và scheduled. Stages, phase gates, stopping conditions.

🤖
Agency Layer

Cách model ĐƯỢC PHÉP hành động. Action substrates (code execution, browser), planner-verifier structures, reviewer roles.

⏱️
Runtime Layer

Cái gì xảy ra khi work unfold theo thời gian: context assembly, memory, compaction, checkpoints, retries, budgets, trace collection. Đây là nơi long-horizon behavior thành công hoặc sụp đổ.

Open Questions

Single agent hay multi-agent tốt hơn? ▼

Chưa rõ liệu single general-purpose coding agent hay specialized agents (testing agent, QA agent, cleanup agent) cho performance tốt hơn. Chuyên biệt hóa có thể giỏi hơn ở sub-tasks nhưng tạo coordination overhead.

Harness coverage — đo thế nào? ▼

Nếu sensors không bao giờ fire, đó là dấu hiệu chất lượng cao hay cơ chế detection không đủ? Cần metric tương tự code coverage / mutation testing cho harness quality.

Generalize beyond coding? ▼

Các bài học từ coding agent harness có thể áp dụng cho scientific research, financial modeling, và các long-running agentic tasks khác — nhưng patterns cụ thể chưa được validate.

Harness coherence khi scale? ▼

Khi harness grow với guides và sensors, làm sao giữ chúng in sync, không contradict nhau? Khi instructions và feedback signals point khác hướng, agent trade-off thế nào?

Natural-language harnesses? ▼

Nghiên cứu đang khám phá việc externalize harness pattern logic thành executable representations — viết harness bằng ngôn ngữ tự nhiên, compile thành runnable constraints. AutoFlow, FlowAgent, AgentSpec đang tiến theo hướng này.

Competitive Advantage đã dịch chuyển

Khoảng cách giữa available models đang thu hẹp. Khoảng cách giữa harness quality giữa các teams đang MỞ RỘNG. Lời khuyên từ toàn bộ lĩnh vực: đầu tư vào harness, không phải vào model.
